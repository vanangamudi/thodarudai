* After conversing with LLM
- Goal: Semi-manual tokenization for Tamil (agglutinative) by splitting long words into meaningful smaller words, preserving multiple possible splits for later expert review.
- Workflow needs:
  - Explore corpus by related forms (shared prefixes/suffixes).
  - Focus on longest words first to avoid redundant work.
  - Interactive editing in Emacs with full power (regex replace, multiple cursors, etc.).
  - Version control all manipulations/decisions.
  - No reliance on statistical methods initially; rules and insights drive splits.
  - Same word can appear in multiple batches to reflect evolving insights.
- Technical constraints:
  - No database.
  - Prefer plain text (TSV/custom), avoid JSON/JSONL unless unavoidable.
  - Keep data loading efficient: run a long-lived process so we don’t reload on every query.
  - Use local sockets instead of HTTP.

Design we agreed on
- Plain-text artifacts (TSV):
  - data/wordlist.tsv (not necessarily versioned)
    - Columns: word, freq, glen (grapheme length, precomputed).
  - batches/*.tsv (versioned; fully editable in Emacs)
    - Columns: word, freq, glen, splits, status, notes.
    - splits: plus-separated parts with optional alternatives separated by “ || ” (e.g., இள+ஞ்+சிவப்பு || இளஞ்+சிவப்பு).
    - Escaping: \+ \| \\ (and keep single-line cells).
    - status: todo | accepted | ambiguous | rejected | draft.
  - ledger/splits-ledger.tsv (append-only, versioned)
    - Columns: timestamp, batch, word, status, splits, notes.
    - Duplicates allowed (same word can appear multiple times across batches).

- Long-lived local server (Unix domain socket; no HTTP):
  - Socket: run/tamil_words.sock (UTF-8).
  - Loads wordlist.tsv once; builds in-memory indices:
    - by_len: sorted by glen DESC, then freq DESC, then word ASC.
    - lex: sorted by word ASC (fast prefix range).
    - rev_lex: sorted by reversed(word) ASC (fast suffix range).
  - Loads ledger to maintain accepted_words set (latest status=accepted by file order).
  - Single-threaded (can add file locks for ledger writes if needed).

- Simple line-based socket protocol:
  - QUERY prefix=… suffix=… min_len=… limit=… offset=… exclude_accepted=0|1 regex=…
    - Returns TSV body with header: word, freq, glen, splits, status, notes.
    - Order: glen DESC, freq DESC, word ASC.
  - COMMIT batch=… rows=N
    - Followed by N-lines TSV (same header). Appends to ledger; updates in-memory accepted set.
  - RELOAD what=ledger|wordlist|both
    - Reloads files and rebuilds in-memory state.
  - STATS
    - Returns counts and mtimes.

- Emacs usage (plain TSV buffers):
  - Run QUERY to open a TSV buffer; edit freely (regex, multiple cursors, etc.).
  - Save buffer to batches/*.tsv for VCS.
  - Run COMMIT from the buffer to append edits to ledger.
  - Defaults focus on longest-first; exclude_accepted enabled to reduce redundancy.

- Redundancy control and ordering:
  - Default queries sort longest-first; set high min_len and exclude accepted words by default.
  - Duplicates across batches are permitted; ledger is append-only.

- Safeguards and validation:
  - Validate TSV headers and status values on COMMIT.
  - Create directories as needed (run/, ledger/).
  - Optionally lock ledger file during append if multiple clients appear.
  - Error responses are explicit (ERR …) without crashing the server.

- Future extensions (optional):
  - exclude_parts: skip words that already appear as parts in accepted splits.
  - group_by_affix in TSV output to cluster related forms.
  - Add top-of-buffer comment block explaining query parameters (as TSV comments starting with #).
