* Project Documentation
This document describes the overall architecture of the project. The project is divided into three main parts: the backend (a FastAPI server), a web frontend (a React app built with Vite), and a desktop client built with PyQt5 (the Qt GUI). Many modules share common utilities and storage abstractions so that they can work with either file‐based storage or a SQLite database.

* Backend Components
The backend is implemented in ~webui1/app.py~. It is a FastAPI application that listens on port 8000 and exposes multiple REST endpoints. Key endpoints include:
- =QUERY=: accepts query parameters (prefix, suffix, regex, negative filters, length spec, limit, curated ratio) and returns a list of words with their metadata (word, frequency, grapheme length). It uses the word index and curation modules.
- =COMMIT=: accepts edited rows (as JSON) and an optional batch name, writes the batch file via the storage layer, and then appends the same data to a ledger.
- =SUMMARY=: recomputes statistics such as the total number of words, curated counts, and length distribution.
- =REMINDERS= (GET and POST): used to read reminder words and update them.
- =GENERATE_BATCH_NAME= and =HEALTH=: provide batch name generation and a diagnostic summary respectively.

The backend relies on helper functions found in ~tools/curation_core.py~ and ~tools/storage.py~. It also loads the word index (via ~tools/word_indexer.py~) and the curated index (via ~tools/curation_index.py~). File paths for the word list, batches, ledger, and reminders are computed by ~tools/profile.py~.

* Storage Abstraction
The module ~tools/storage.py~ defines a common abstraction ~StorageBase~. Two implementations exist:
- ~FileStorage~ uses TSV files for batch, ledger, and reminders.
- ~SqliteStorage~ uses a SQLite database. It configures ~PRAGMA journal_mode=WAL~, ~PRAGMA synchronous=NORMAL~, and sets a busy timeout so that concurrent access is supported.
Both the backend and the Qt GUI use this abstraction so they can switch storage modes simply by configuration.

* Word Index and Curation
There are several modules dedicated to indexing and managing words:
- ~tools/word_indexer.py~ implements a list‐based word index. It loads a TSV word list (with columns *word*, *freq*, *glen*) and supports queries with prefix and suffix filtering, as well as using helper filters from ~tools/curation_core.py~.
- ~tools/trie_word_indexer.py~ provides an alternative trie‐backed index. This module uses the ~OnDiskTrie~ class from ~chorkilai/trie.py~ to build separate forward and reverse tries for very fast prefix and suffix lookups.
- ~tools/curation_core.py~ supplies utility functions that parse length specifications, combine new and curated rows, filter eligible candidates, and build TSV lines.
- ~tools/curation_index.py~ implements a Bloom filter–based curated index. It tracks which words have been curated (via batches) and incrementally updates the curated count without scanning the entire word list.
In the Qt GUI and backend, the chosen word index (either list-based or trie-based) is queried and combined with the curated index to produce the final set of words for display and commit.

* Data Export and Batch Processing
The project includes scripts for building and exporting data:
- ~tools/build_word_index.py~ creates the TSV word index from one or more source files.
- ~tools/export_batches_dataset.py~ aggregates curated batches into a dataset TSV for tokenizer training.
- ~tools/export_dataset.py~ (currently empty or minimal) is intended for similar export tasks.
Batches are written in TSV format. The ledger (which records every batch commit, including summary records) is updated via the storage API.

* Tokenizer Training
The module ~tools/train_tokenizer.py~ implements training for a sequence-to-sequence model using PyTorch. It loads a dataset (TSV file) and builds vocabulary mappings from tokens. It then sets up an encoder and decoder and trains a simple model. The tokenizer training uses a dataset generated by exporting curated batches.

* Tamil Phonetic Transliterators
The module ~tools/tamil_phonetic.py~ implements the transliteration engine. It defines mappings for standalone vowels, vowel diacritics (to attach to pending consonants), consonant tokens, and the pulli, which suppresses the inherent vowel. The function ~transliterate(text)~ processes romanized Tamil input (greedily matching tokens) and produces Tamil script. This module is used by the Qt GUI for interactive input.

* Frontend Components (Web UI)
The web frontend is built with React and bundled with Vite. Key files and their roles include:
- ~webui1/frontend/package.json~ and ~vite.config.js~ define the project and build configuration.
- ~webui1/frontend/src/main.jsx~ is the entry point where the top-level React ~App~ component is rendered.
- ~webui1/frontend/src/App.jsx~ serves as the main container and loads the ~QueryForm~ component.
- ~webui1/frontend/src/QueryForm.jsx~ provides a complete form for entering query parameters, viewing results, performing find/replace operations, and committing edits.
- UI subcomponents include:
  - ~FiltersPanel.jsx~: fields for prefix, suffix, regex, and their negations.
  - ~OptionsPanel.jsx~: controls for length specification, limit, curated ratio, and buttons to query, commit, or load summary.
  - ~FindReplacePanel.jsx~: handles find-and-replace operations on the splits column.
  - ~ResultsTable.jsx~: displays results in a table.
  - ~SummaryPanel.jsx~: shows summary statistics.
  - ~FooterInfo.jsx~: displays UI build timestamp and health data from the backend.
- API calls are wrapped in ~webui1/frontend/src/api.js~ using ~axios~. The frontend communicates with the backend REST API (running in ~webui1/app.py~) over HTTP.

* Qt GUI Components
The desktop client is implemented in ~tools/gui_qt_app.py~ using PyQt5. It provides a workflow similar to the web UI, but in a native desktop window.
Key elements include:
- ~PhoneticLineEdit~: a custom QLineEdit subclass that dynamically transliterates input using ~tools/tamil_phonetic.py~. It maintains separate buffers for committed text, composition, and trailing text.
- ~TableCellLineEdit~ and ~TableEditDelegate~: support in-cell editing in the results table.
- ~MainWindow~: the main application window. It assembles panels for entering query parameters (prefix, suffix, regex, and their negative filters), a table for displaying results, and a summary panel for statistics. It loads the word index (using ~tools/word_indexer.py~ or the trie‐based index if selected) and a curated index (via ~tools/curation_index.py~ or ~CuratedIndexDB~ if using SQLite storage).
- The UI supports interactive query execution, find/replace actions, sorting, and opening new windows. It logs all UI events to a log file defined by ~UI_LOG_PATH~.
- Like the backend, the Qt client uses the storage abstraction from ~tools/storage.py~, allowing both applications to share the same batch, ledger, and reminder data.

* Intercomponent Communication and Independence
- The backend (FastAPI server) and the frontend (React app) are independent applications that communicate over HTTP using the REST API. The endpoints in ~webui1/app.py~ form the contract between them.
- The Qt GUI is a separate desktop client. It does not communicate with the backend via HTTP but accesses the same file system (or SQLite) for data storage (word-index TSV, batch files, ledger, reminders). Thus, the Qt GUI and backend can operate on the same underlying data.
- The storage layer (~tools/storage.py~) is central to both the backend and the Qt GUI. Switching between file-based storage and SQLite is done through a configuration parameter.
- The word indexing and curation components (~tools/word_indexer.py~, ~tools/trie_word_indexer.py~, ~tools/curation_core.py~, and ~tools/curation_index.py~) are independent modules. They provide an API to search and rank words and to mix curated words with new ones. Both the backend and the Qt GUI rely on these modules.
- The transliteration module (~tools/tamil_phonetic.py~) is self-contained and used by clients (primarily the Qt GUI and parts of the frontend for on-the-fly display).

* Summary of Flows
- When a user enters query parameters in either the web UI or the Qt GUI, the input is normalized (for example, converting a romanized prefix into Tamil may occur on the fly in the Qt GUI using ~PhoneticLineEdit~).
- The query is passed to the word index (either list-based or trie-based) which returns candidate words.
- The curated index is consulted to mix curated words with new words according to a curated ratio.
- The combined result is returned to the user: the backend returns a JSON response (or HTML if requested) while the frontend and Qt GUI display the results in a table.
- Edits (e.g., corrections to splits) are committed by triggering a commit. The storage abstraction then writes a batch file and updates the ledger (and possibly the curated index).
- The backend exposes endpoints for refreshing summary statistics, managing reminders, and reporting health status. The Qt GUI has similar capabilities operating on the shared file/system storage.

* Conclusion
This project is organized into loosely coupled modules. Core functions (word indexing, curation, storage, and transliteration) are implemented in the ~tools/~ directory and reused by both the backend and the desktop client. The web frontend communicates with the backend exclusively via REST, independent of the Qt GUI. The shared storage abstraction allows multiple interfaces to operate on the same underlying data.
