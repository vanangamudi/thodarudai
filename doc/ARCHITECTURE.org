#+author: Project maintainers

* Introduction
- Goal: build a multi-user platform for manually segmenting Tamil words into two parts, and to navigate a corpus with fast prefix, suffix, and regex queries.
- Context: original Qt tool with file storage; target is a Web UI backed by databases (SQLite or Postgres). Keep Qt operable via shared modules and storage abstraction.
- Scope: system design, canonical data model, API, storage backends, query engine, front-ends, migration plan, testing, and operations.

* Functional requirements
- Corpus ::
  - Immutable set of words with attributes =freq= and =glen= (grapheme length).
  - Invariant: =glen = ari.length(word)=.
- Segmentation ::
  - Multiple user-contributed binary splits per word.
  - Each split is at a grapheme boundary: =1 <= split_pos < glen=.
  - Text cache: =left_text= and =right_text= such that =left_text + right_text = word= at the grapheme boundary.
  - Notes are free-form.
- Query ::
  - Prefix, suffix, and regex filters.
  - Negative filters: =prefix_not=, =suffix_not=, =regex_not=.
  - Length filter via =length_spec= (examples: "8-", "4-9", "7").
  - Ranking order: =glen DESC, freq DESC, word ASC=.
  - Curated mixing: blend new (no segmentations yet) with curated (has segmentations) by ratio.
- Multi-user ::
  - Writes attributed to an =actor_id= (stored as =profile=).
  - Reads default to global (all users), optionally scope to "me" or "actor:NAME".
- Reminders ::
  - Per-user set of words to revisit.
- Summary ::
  - Totals (words, curated distinct, remaining distinct, curation entries).
  - Length distributions for curated and remaining.
- History ::
  - List all segmentations per word, newest first.
- Observability ::
  - Request logs, health endpoint, optional UI event logs.

* Canonical data model (DB v2)
- Tables (SQLite and Postgres; Postgres may use =pg_trgm= for performance):

#+begin_src sql
-- words: canonical corpus
CREATE TABLE IF NOT EXISTS words (
  word TEXT PRIMARY KEY,
  freq INTEGER NOT NULL,
  glen INTEGER NOT NULL
);

-- segmentations: truth source for curation
CREATE TABLE IF NOT EXISTS segmentations (
  id BIGSERIAL PRIMARY KEY,              -- INTEGER AUTOINCREMENT in SQLite
  word TEXT NOT NULL REFERENCES words(word) ON DELETE CASCADE,
  profile TEXT NOT NULL,                 -- actor_id
  split_pos INTEGER NOT NULL,            -- 1..glen-1
  left_text TEXT NOT NULL,
  right_text TEXT NOT NULL,
  notes TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS seg_word_created ON segmentations(word, created_at DESC);
CREATE INDEX IF NOT EXISTS seg_profile_word ON segmentations(profile, word);

-- ledger: append-only commit rows (kept for parity with FS)
CREATE TABLE IF NOT EXISTS ledger (
  id BIGSERIAL PRIMARY KEY,            -- INTEGER AUTOINCREMENT in SQLite
  ts TIMESTAMPTZ NOT NULL,
  batch TEXT NOT NULL,
  rec_id TEXT,
  word TEXT,
  splits TEXT,
  notes TEXT,
  profile TEXT
);

-- reminders: per-user bag of words
CREATE TABLE IF NOT EXISTS reminders (
  profile TEXT NOT NULL,
  word TEXT NOT NULL,
  notes TEXT,
  updated_at TIMESTAMPTZ DEFAULT now(),
  PRIMARY KEY(profile, word)
);
#+end_src

- Invariants ::
  - =1 <= split_pos < glen= for the corresponding =word=.
  - =left_text + right_text = word= at a grapheme boundary.
  - Multiple segmentations allowed per (profile, word) and globally.

- Views (optional convenience)

#+begin_src sql
-- latest segmentation per word (global)
CREATE VIEW v_latest_segmentation_global AS
SELECT DISTINCT ON (word)
  word, profile, split_pos, left_text, right_text, created_at
FROM segmentations
ORDER BY word, created_at DESC, id DESC;

-- latest per profile
CREATE VIEW v_latest_segmentation_by_profile AS
SELECT DISTINCT ON (profile, word)
  profile, word, split_pos, left_text, right_text, created_at
FROM segmentations
ORDER BY profile, word, created_at DESC, id DESC;
#+end_src

* Indexes and performance notes
- Postgres ::
  - Enable =pg_trgm=:
    ~CREATE EXTENSION IF NOT EXISTS pg_trgm;~
  - Trigram index for =LIKE= and regex:
    ~CREATE INDEX IF NOT EXISTS idx_words_word_trgm ON words USING gin (word gin_trgm_ops);~
  - Optional suffix optimization:
    ~CREATE INDEX IF NOT EXISTS idx_words_rev ON words ((reverse(word)));~
- SQLite ::
  - B-tree indexes on =word= and =glen=.
  - Optional FTS5 mirror for advanced regex (future).

* Query engine
- Inputs
  - =prefix=, =suffix=, =regex=, =prefix_not=, =suffix_not=, =regex_not=, =length_spec=, =limit=, =curated_ratio=.
  - Optional =curated_scope= with values "all" (default), "me", "actor:NAME".
- Normalization
  - Implemented in =tools.curation_core.normalize_query_fields= and =parse_length_spec=.
- Ranking
  - Canonical sort key: =(-glen, -freq, word)= which maps to SQL =ORDER BY glen DESC, freq DESC, word ASC=.
- Curated mixing
  - Current DB mode: =query_index()= in =SqliteStorage= and =PostgresStorage= applies positive filters (prefix, suffix, regex, length) and canonical ranking. Negative filters and curated mixing are applied in the app/GUI layer after fetching rows.
  - FS or in-memory mode: sample curated vs new in Python using =mix_curated()=.
- Probe size
  - When client-side mixing is needed, probe =min(limit * 5, max(limit + 500, 5000))=.

* API design
- POST =/api/query=
  - Body: fields listed above.
  - Returns: ~{ results: [[word, freq, glen], ...], elapsed_ms: N }~ or HTML table when Accept is text/html.
- POST =/api/commit=
  - Body: ~edited_rows~ ([[id, word, splits, freq, glen, notes] ...]), optional ~batch~.
  - DB: parse =splits= into (=left_text=, =right_text=, =split_pos=) and call =add_segmentation()= for each row. Write an optional ledger event.
  - FS: write TSV batch and append ledger.
  - Returns: counts and storage paths or tokens.
- GET =/api/segmentations?word=... [&scope=...] =
  - Returns the full history of segmentations for the word, newest first, optionally scoped to "me" or "actor:NAME".
- GET =/api/summary[?scope=all|me|actor:NAME]=
  - Returns totals and length distributions, default is global.
- GET/POST =/api/reminders= and GET =/api/reminders/results=
  - Per-user list management and lookup against the corpus.
- POST =/api/generate_batch_name=
  - Utility based on =prefix=, =suffix=, and =length_spec=.
- GET =/api/health=
  - Build tag, app mtime, counts, storage info.

* Migration plan
- Phase 1: DB enablement ::
  - Add =segmentations= table and minimal =get_latest_splits()= in DB stores.
  - Implement =add_segmentation()= and =list_segmentations()= in =SqliteStorage= and =PostgresStorage=.
  - In backend, =/api/commit= uses =add_segmentation()= in DB modes; FS path unchanged.
  - Keep ledger for events, not as truth.
- Phase 2: Auth and scoping ::
  - Add minimal auth, set session =actor_id=, wire it to Storage.profile.
  - Expose =curated_scope= in queries and =scope= in =/api/summary= and =/api/segmentations=.
- Phase 3: Query optimization ::
  - Enable =pg_trgm= and indexes in Postgres; consider SQLite FTS5 mirror.
  - Push more filtering into SQL in =query_index()= (negative filters, curated mixing).
- Phase 4: UX and insights ::
  - Curated badges and per-word history in Web UI.
  - Conflict awareness in UI (warn when a newer segmentation exists).
- Phase 5: Deprecations ::
  - De-emphasize FS ledger as source of truth where DB is available.
  - Optional deprecation of list indexer in favor of trie or DB-only for production.

- Auth model :: minimal session vs external provider.
